# Поднятие своего приложения на кластере Kubernetes с помощью Ansible


## Кратко про проект
Есть фронт(сгенерирован полностью кодексом по одному запросу), есть 3 микросервиса (авторизация, юзер, друзья), первые два используют постгрес, третий neo4j (графовую базу данных). Код в основном писался руками, потом чтобы не заниматься нудной прокидкой трейсов/логов в каждую первую функцию был также применен кодекс.

Код не оптимизирован, и с вероятностью очень много процентов плохой, что в целом пока не является проблемой ибо за его полировку я пока не садился. Основной целью было именно тест ансибла, кубера и в будущем ОпенСтака.

## Подготовка

### Доступные ресурсы
Конечно хотелось попробовать бы поднять HA кластер, с 3 мастер и 4 воркер нодами, однако в связи с сильным ограничением на ресурсы: 

* 6(12) core CPU
* 16GB RAM
* ~150-180GB Storage

Было принято вынужденное, но героическое решение отказаться от всех возможных best-practice или best-strategies (HA, dns, nfs) по развертыванию кластера и оставить все сугубо в учебных/тестовых рамках.

### Топология
Исходя из доступных ресурсов, топология проекта будет МАКСИМАЛЬНО сжатой:

* 1 master нода (OS: Ubuntu Server LTS 22.04.4, 2 core CPU, 2GB RAM, 25GB Storage)
* 2 worker ноды (OS: Ubuntu Server LTS 22.04.4, 2 core CPU, 2GB RAM, 25GB Storage каждая)

Согласно документации master нода требует больше минимальных ресурсов для запуска чем worker нода, но в конечном итоге worker ноды будут использовать гораздо больше ресурсов чем master ноды. 



### Недостатки
Несмотря на то, что кластер полностью функционален и работает - такое малое количество ресурсов, отсутствие реплик деплойментов и отсутствие HA очень сильно бьет по смыслу реализации Кубера.

Количество реплик каждого Deployment или StatefulSet ограничились значением 1 (т.е поднимается по одному экземпляру).

Еще один важный нюанс: поднимать БД в кластере Кубера - идея не очень хорошая. Контейнеры - эфемерны, их состояние не сохраняется после перезапуска/удаления. Что немного противоречит смыслу существования базы данных. В самом Kubernetes есть технологии для решения ситуации, но из всех услышанных мной рекомендаций лучше держать их вне кластера.

Но, еще раз, ресурсы ограничены - поднимать отдельную ВМ для БД возможности нет, поэтому в моем случае БД поднимается внутри кластера.

При расширении пула ресурсов я бы сразу добавил следующее:
* HA (+2 мастер ноды к уже имеющейся, количество должно быть нечетным)
* Также 2 воркера
* Увеличил количество реплик каждого приложения
* Поднял бы отдельный DNS, реализовал бы local node DNS
* Поднял бы БД на отдельных машинах вне кластера
* Попробовал бы поменять CNI на Cilium (в целом ради интереса, но в любом случае он мощнее)

## Виртуальные машины

Выше было указано что используется Ubuntu Server LTS 22.04.4
Весь процесс установки разбирать особого смысла нет, но важно подметить некоторые детали:

### Данные в ВМ
![regist](https://i.ibb.co/TDbJTfxf/photo-2026-04-22-13-36-38.jpg)

В процессе установки обязательно дать ВМ разное имя сервера, иначе: если у вас, например, будет 2 машины с worker - одна из них проста будет не видна.

username (aka hostname) и пароль будет удобно сделать одинаковыми для всех ВМ. В моем примере - хостнейм мастер ноды и воркер нод разный (master, worker, worker), но пароль одинаковый.

Если пароль будет разный, то --ask-become-pass при запуске Ansible плейбуков не справится, но ведь всегда можно просто отключить пароли вовсе.

На всех ВМ:
```
sudo visudo
%username% ALL=(ALL) NOPASSWD:ALL
```
### OpenSSH
При установке ОС не забудьте поставить галочку на пунтке openssh.
Если забыли: 
``` 
sudo apt update
sudo apt install openssh-server
```
Проверьте запущен ли, и у запустите если нет:

```
sudo systemctl status ssh
sudo systemctl start ssh 

sudo systemctl enable ssh // если нужен автозапуск
```

### Сеть и IP
Если на каждой ВМ показывает одинаковый IP: 
```
Создаем новую сеть:
VirtualBox -> Network -> Host-Only Network -> Create

После добавляем только что созданную сеть на каждую ВМ:
VM Settings -> Network -> Adapter 2

Attached to:      Host-Only Adapter
Name:             vboxnet0 // только что созданная
Promiscuous Mode: Allow All

После чего уже на самой ВМ:
sudo nano /etc/netplan/*.yaml

Туда, к уже существующему адаптеру нужно добавить:
enp0s8			// дефолтное значение второго адаптера в VirtualBox
  dhcp4: true   // если использовали другой: проверьте через ip a


sudo netplan try // чек на возможные ошибки, откат если есть
sudo netplan apply
```

Это должно исправить проблему. Не забудьте записать каждый username и IP виртуальной машины, это позже пригодится.

### Диск

При установке ваше выделенное пространство делится на три раздела, и бывает, что ОС не использует все доступное хранилище.

Из-за этого, если вы выделили мало памяти, в дальнейшем когда вы будете импортировать образы в ваш CRI - может так случится, что он упрется в свой лимит 85% от всей доступной памяти и начнет удалять старые образы. Пока вы будете импортировать новые, он будет удалять те которые вы импортировалив минуту назад.

Поэтому нужно сделать так чтобы ОС использовала всю память:

```
lsblk // смотрим название раздела которое использует наша ОС

sudo growpart /dev/sda 3
sudo pvresize /dev/sda3
sudo lvextend -r -l +100%FREE /dev/mapper/ubuntu--vg-ubuntu--lv

df -h // проверка
```

![lsblk](https://i.ibb.co/0yrT21q0/photo-2026-04-22-13-30-51.jpg)

### SSH ключи
Нужно записать с каждой ВМ её IP и username: 
```
hostname -I // ip
whoami // username
```
На основной машине (не ВМ) выполняем следующее:
```
ssh-keygen -t ed25519 // -c метку опционально

```

### Образы
Желательно заранее закинуть образы на ВСЕ виртуальные машины. Так как никто кроме самого Kubernetes не знает на какой именно ноде он запустит под - образы должны лежать на всех (хотя бы) worker нодах. Если taints вашей master ноды сконфигурирован так, что в случае какой-то проблемы поды будут должны подняться на ней, то - и на эту ноду тоже нужно закинуть образы.

```
docker pull ... & docker build ...
docker save imageName imageName.tar

scp *.tar username@ip:/home/username/
```

## Настройка

### hosts.ini

В файл нужно записать IP адрес виртуальных машин, username, и указать где лежит ssh ключ. Пример файла можете найти в `ansible/inventory/hosts.ini.example`

Там же необходимо разделить наши ВМ на группы - `masters` и `workers`. Это нужно чтобы в будущем выполнять необходимое команды только на нужных машинах.

После этого можете проверить, успешно ли проходит коннект:
`ansible all -m ping`. Если просит пароль, то к этой и в дальнейшем при запуске плейбука добавьте флаг `--ask-become-pass`.

### group_vars

Все переменные которые будут использованы находятся в `group_vars/all.yml`. Чтобы не копипастить одни и те же значения в разных файлах. 

## Плейбук
Наш первый (и единственный) плейбук находится в папке - `playbooks/k8s_install.yml`. Всего в нем три этапа: 
- Подготовить все ноды
- Инициализироовать мастер ноду
- Подсоединить воркеров 

Для этого мы и создавали разные группы, так как первый этап выполняется на всех виртуальных машинах, второй только на ВМ которая будет master-ом, а третий этап на оставшихся двух.

## Подготовка всех нод
Прежде чем начать, нужно подготовить ВМ к тому, чтобы они могли быть полноценными нодами в кубере. Здесь лишь описано что, зачем и почему происходит, все команды в деталях можете посмотреть напрямую в файлах ролей - `ansible/roles`.

### k8s_prerequisites
Перед тем чтобы поставить кубер, мы должны сделать следующее:
- Отключить swap - чтобы система не использовала диск, если оперативная память кончится. Эта выделенное пространство гораздо медленнее чем RAM, но чисто на бумаге она будет такой же. Поэтому Кубер будет думать что у него еще есть ресурсы, когда на деле все начнет замедлятся.
- Подключаем нужные модули ядра. Для containerd и сетевых правил Кубера.
- Настроить параметры ядра для маршрутизации трафика.

>Важный момент, так как в примере не реалзиуется HA, дополнительно НЕ задается ipv4.ip_nonlocal_bind - поскольку этот параметр нужен лишь для занятия виртуального айпи который будет ответственен за подключение к разным мастер нодам. В данной задаче у нас лишь одна мастер нода, поэтому мы благополучно забиваем на это.

- После чего нужно установить базовые пакеты. 

> Важная деталь 2. Если у вас другая ОС на машинах, также обязательно НУЖНО отключить SELinux и Firewall на каждой из них. На выбранной мной (Ubuntu Server 22.04.4 по умолчанию и то, и другое отключено поэтому этих пунктов в плейбуке нет).

### containerd
Чтобы запускать контейнеры(поды) нужно на каждую ВМ установить container runtime. Нужно создать дефолтный конфиг и указать, чтобы оба, kubelet и containerd, должны использовать одинковый способ управления ресурсами - systemd. Kubelet и так по умолчанию его использует, поэтому мы подгоняем настройку containerd под него - `SystemdCgroup=true`

### kubeadm
Так как нужных нам пакетов (kubeadm, kubectl, kubelet) нет в подключенных репозиториев (по крайне мере в Ubuntu и Debian). Нужно добавить репозиторий от Кубера в систему - но перед этим нужно взять также apt key с репозитория Кубера, чтобы система знала что файлы настоящие и никак не поврежедены (подмена итд). После чего можно спокойно установить kubeadm, kubelet, kubectl через `apt install`.

После чего нужно поставить пакеты на холд, чтобы случайный `apt upgrade` не смог обновить их.

Даем информацию kubelet о IP адресе машины, настраиваем crictl чтобы он использовал containerd. Сам crictl установился чуть раньше, как зависимость kubeadm.

После чего включаем сервис kubelet на всех машинах через `systemctl enable kubelet`.

### local_images
Так как в производственной среде скорее всего нельзя будет пуллить образы из интернета - нам надо закинуть наши образы в неймспейс кубера в containerd. В самом начале мы заранее закинули архивы с нашими контейнерами на виртуальные машины.

Как и говорил ранее, если ваша мастер нода не расчитана на поднятие подов в экстренном случае можете перенести этот процесс в роль `k8s_workers`, и образы изначально на нее можете не передавать.

В общем, роль просто выполняет команду `ctr -n k8s.io images import /home/*username*/*imagename*.tar` n-количество раз.

> Интересный факт, образ фронта пересобирался примерно 7-8 раз прежде чем смог нормально запуститься, поэтому вам возможно стоит иметь быстрый доступ к ВМ-кам - закинуть один образ на 2-3 машины гораздо быстрее чем ждать пока Ansible повторно дойдет и начнет делать это сам.

>Более интересный факт, только в процессе написания было замечено что этой роли в принципе не существует. Видимо при изначальной настройке я забыл, что закидывал образы вручную. Поэтому просто надеюсь что оно работает :)

## Инициализация master node
Теперь переходим к работе с конкретной ВМ котороая станет master нодой. 

### k8s_master
Запускает kubeadm init на нашей мастер ноде, указывая CIDR для сервисов и подов, а также IP адрес который будет использовать API сервер (адрес этой же ноды).

В этот момент создается control plane - т.е вышеупомянутый API-сервер, планировщик, контроллер и база данных etcd. 

Назначаем IP-адрес который будет использоваться как DNS и перезапускает сервис kubelet.
`kube adm init` создал `admin.conf` в директории `/etc/kubernetes/` - что не очень удобно поскольку находится в рут директории. 

Нужно создать директорию не в рут директории и скопировать туда конфиг. После чего генерирует команду для присоединения воркеров и сохраняет ее на локальной машине откуда запускался Ansible. Чтобы потом при подсоединении воркеров использовать ее повторно.

> На самом деле, эта команда для подключения генерируется и выводится при `kube adm init`, но так как это немного муторно вытаскивать ее из потока данных - мы просто пересоздаем ее. Стоить также заметить, что у этой команды есть TTL, и по дефолту она живет 24 часа. После чего токен который используется как один из флагов перестает быть действительным и команда больше не работает.

### cni
Для простоты реалзации, в качестве CNI я выбрал Flannel. Но создал для этого отдельную переменную, если есть желание поменять на Calico, просто поменяйте значение.


Важный момент, перед тем как начать это делать надо подождать пока поднимется под API сервера и только после этого начинать выполнение команд.

Это один из самых простых этапов потому что мы просто скачиваем готовый манифест и буквально просто создаем под по нему, после чего запускаем `create -> apply`

## Подключение workers
Теперь, когда master нода готова, можем переходить к настройке других нод.

### k8s_workers
Выполняет ранее сохранную команду для присоединения к мастер ноде на двух машинах. Также указывает в сервисе kubelet IP адрес DNS и перезапускает его.

## Команды для тестов
Решил написать этот раздел как напоминание для себя что мне помогало решать проблемы:

`kubectl get [*nodes/pods*] -n [*namespace*] -l [*label*]` - проверка общего состояние подов, тег лейбл упрощает жизнь очень сильно, если нужна общая картина то флаг `-A`.

`kubectl describe -n [*namespace*] pod [*podname*]` - помогает если какой-то из подов застревает в `CrushLoopBackOff` и даже не запускается, а значит и логов нет.

`kubectl logs [*podname*]` - спонсор ребилда образа 8 раз подряд, просто позволяет смотреть логи пода.

Много других полезных команд, прям все перечислить очень трудно `kubectl get <type>`, `kubectl rollout restart`, `kubectl top <type>` и другие.

## Манифесты

### Начало и порядок запуска
Прошлым шагом - мы лишь настроили сам кластер и подготовили его к эксплуатации. Однако, нужно ещё задеплоить туда приложение.

Для начала стоить сказать, что Кубер, если ему не сказать в каком порядке - начнет просто читать файлы по порядку, а это значит, что некоторые core компоненты могут быть применены уже после других, зависящих от них.

> Например, возьмите ситуацию что `deployment`, который должен подняться в определенном `namaspace`, применяется первее, чем сам `namespace`.

Чтоб такого не происходило - сущестует специальный тип манифестов - Kustomization. Там вы можете описать что и в каком порядке должно будет применяться.

Деплой теперь будет происходить с другим флагом:

Было: ```kubectl apply -f name.yaml```

Стало: ```kubectl apply -k kustomization.yaml```

### Структура

Теперь рассмотрим как выглядит каждая папка. Я постараюсь какие-то моменты подкрепить кодом конфига, но так как если пояснять каждый момент это будет очень долго нудно и неинтересно, а это уже в целом неинтересно читать - все файлы можете найти в репозитории в папке `k8s`.

#### k8s/core
Самые основные компоненты нашего деплоя - без них дальше никуда. В моем проекте это - namespace и secrets.

* namespace - именно в этом абстрактном пространстве будут запускаться все поды приложения, даже если они на разных нодах они все равно будут находитсья в этом неймспейсе. Не знаю с технической точки зрения, но в моем нынешнем понимании можно относиться к этому как к еще одному лейблу.

* secrets - это переменные, пароли, логины для дб и остального. Сразу скажу, что реализованный вариант абсолютно небезопасный, и так делать, наверное, лучше не стоит. Но так как все делается для локального теста в целом мне и так подошло.

#### k8s/network
В этом файле, применяются стандартные/шаблонные полтики, они будут применятся изначально ко всем подам в этом пространстве имен. Всего в папке их две:

- deny-all - запрещает любой входящий/исходящий трафик для подов в кластере. Вайтлист всегда сделать проще, чем невозможный блеклист.
- allow-dns-engress - после предыдудщей политики ДНС тоже перестал работать, поэтому мы разрешаем весь исходящий трафик на UDP/TCP порт 53 всем подам с лейблом kube-system.

На данный момент если оставить все так - сервисы между друг другом общаться не смогут, поскольку любой трафик сейчас запрещен. Дальше, когда мы будем описывать уже поды приложений - мы будем давать им собственную network policy и прописывать там куда они могут обращаться и откуда могут получать трафик.

#### k8s/storage
Для будущего хранения данных БД нужно, чтобы под, когда запустится, создал на той же ноде директорию, куда будет сохранятья файлы - честно плохое решение, поскольку если под упадет, то остается только надеяться на удачу что он будет на той же ноде где и данные, а иначе они просто будут недоступны ему. 

Как костыль решения можно предложить залочить ДБ на определенную ноду. Как не костыль - облачные дб.

```
apiVersion: v1
kind: PersistentVolume
metadata:
  name: neo4j-data-pv
spec:
  capacity:
    storage: 5Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: local-storage
  hostPath:
    path: /mnt/petbro/neo4j
    type: DirectoryOrCreate
```

#### k8s/apps 
> ОЧЕНЬ ВАЖНЫЙ АЛЕРТ! В дальнейших пунктах databases/apps, идет описание деплойментов, а в следствии и шаблона пода каждого написанного мною приложения - НО у них всех отсутствует одна важная деталь, а именно - реквесты и лимиты (cpu, ram) самого пода. Несмотря на то, что это опциональная фича, по факту она общепризнана обязательной и запускать что-то без нее - ну смерть приложению. Я это упустил лишь потому, что знал что приложение запустится на тех ресурсах которые я выделил машине, и не крашнется, и я, в количество 1 человека, смогу пользоваться этим. Разберу на примере UserService:

##### deployment
В начале каждого манифеста указываем api версию, тип и метаданные
```
apiVersion: apps/v1
kind: Deployment
metadata:
  name: userservice
  namespace: petbro
```
Далее мы указываем как должен выглядеть наш финальный результат - нам надо указать количество реплик, как деплоймент будет определять ответственнен ли он за под, и конечно, темплейт самого пода.

```
spec:
  replicas: 1 // количество реплик
  selector:
    matchLabels: // будет определение по равенству лейблов
      app: userservice // если есть такой лейбл, то значит его
```

Дальше мы указываем все, что характерно для описания пода (буквально если уже есть готовый под можете вставить просто): имя, образ, политика пулла образа из интернетау, порты, переменные, а также важная часть - пробы, проба на готовность, проба жив ли и проба на готовность (есть еще третья, но здесь она не нужна).

```
  template:
    metadata:
      labels:
        app: userservice
    spec:
      containers:
        - name: userservice
          image: petbro-userservice:latest
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 8081
          env: 
```

```
          readinessProbe:
            httpGet:
              path: /health
              port: 8081
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8081
            initialDelaySeconds: 15
            periodSeconds: 20
```

И опять же, не забудьте добавить requests/limits для вашего темплейта пода: 

```
          resources:
            requests:
              cpu: "100m" // 0.1 cpu 
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
```

##### service
Сервис отвечает за стабильный доступ к подам. Кто-то хочет обратиться к поду -> обращается к сервису этого приложения.
```
spec:
  selector:
    app: userservice // отвечает за все поды с таким лейблом
  ports:
    - port: 8081 // на этот порт другие поды отправляют трафик
      targetPort: 8081 // containerport наших подов
```

> Есть еще nodePort, он используется чтобы можно было получить доступ к сервису из вне. Такое применяется только для фронта, прометеуса и графаны.

##### network policy
Сетевые политики отличаются от приложения к приложению, даже на моем примере - userservice обращается почти ко всем сервисам, в то время как friendservice обращается только к одному-двум. 

Также стандартно задаем базовые значения, указываем какие политики мы будем затрагивать:

```
spec:
  podSelector:
    matchLabels:
      app: userservice
  policyTypes:
    - Ingress
    - Egress
```

Пример редактирования входящего трафика:
```
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: authservice
      ports:
        - protocol: TCP
          port: 8081
```

Пример редактирования исходящего трафика:
```
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: postgres
      ports:
        - protocol: TCP
          port: 5432
```

Также в политику к egress трафику необходимо добавить обращние к kube-system за DNS.

```
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

Таких правил может быть бесконечно много, как уже и говорил ранее, безопаснее запретить все, и потом добавлять нужное в вайтлист.

#### k8s/databases

Всего в проекте используется две базы данных - neo4j и postgres.

Список файлов в папках примерно такой:
- StatefulSet - аналог деплоймента (stateless) для приложений требущих сохранение данных(stateful)
- Network Policy
- Service
- configmap - только для postgres, чтобы в самом начале он создал нужные БД

Разберем на примере neo4j

##### StatefulSet
В целом вы можете посмотреть пример выше, где описывался deployment, он мало чем отличается за исключением изменения значения базовых параметров (apiVersion, kind), также рекомендуется убрать livenessProbe поскольку она слишком критична - ваша база может быть просто загружена, и не успеть ответить, а кубер уже пойдет на нее с крестовым походом, вместо этого нужно добавить пробу на запуск (startUp probe). Также нужно добавить темплейт для volumeClaim, с указанием на ранее созданный local storage, чтобы у каждого пода было место для выгрузки.

```
          startupProbe:
            tcpSocket:
              port: 7687
            initialDelaySeconds: 10
            periodSeconds: 10
            failureThreshold: 30
```

```
  volumeClaimTemplates:
    - metadata:
        name: neo4j-data
      spec:
        storageClassName: local-storage
        accessModes:
          - ReadWriteOnce
        resources:
          requests:
            storage: 5Gi
```

> Важный момент, если ставить больше одной реплики, то надо будет настраивать репликацию данных, чтобы два разных пода не смогли записать одно и тоже значение, или чтобы запись принималась толькой мастер дб, а не слейвом, но у меня 1 реплика и поэтому этого функционала нет. Да и выше упоминал про возомжную нецелесообразность поднятия баз данных в Кубере.

##### Service
Сервис от Deployment-а почти ничем не отличается за исключением одной детали: 
```
spec:
  clusterIP: None
```

Это нужно, чтобы сервис выдавал не случайный IP, или по балансировщику, а чтобы он выдавал конкретный под к которому обращаются. Но в моем сетапе, опять же, реплика 1, поэтому значения это особого не имеет.

##### Network Policy
Никаких отличий от deployment-а, конкретную реалзицию можете увидеть в папке.

#### k8s/monitoring

- configmap
- Deployment
- Service
- Network Policy

Для начала, нам надо создать переменные для Prometheus и Grafana соответственно. В первом случае с каким интервалом и что именно нужно скрапить, а во втором к чему получать доступ.

##### config
```
apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-config
  namespace: petbro
data:
  prometheus.yml: |
    global:
      scrape_interval: 15s
      evaluation_interval: 15s
    // у вас должен быть endpoint /metrics в самом сервисе
    scrape_configs:
      - job_name: prometheus
        static_configs:
          - targets:
              - localhost:9090
```
##### Deployment
Сам конфиг тоже мало чем отличается, лишь добавлением параметра --args с ссылкой на конфиг, и передачу информации из конфиг мапы.
```
      volumes:
        - name: config
          configMap:
            name: prometheus-config
        - name: storage
          emptyDir: {}
```

##### Service
В сервисе прописываем nodePort
```
apiVersion: v1
kind: Service
metadata:
  name: prometheus
  namespace: petbro
spec:
  type: NodePort
  selector:
    app: prometheus
  ports:
    - port: 9090
      targetPort: 9090
      nodePort: 30090
```

##### Network Policy
Никаких изменений по конфигу критично отличных от других нет.

Prometheus: 
- Входящий - разрешить весь (можете ограничить теми сервисами у которых берете метрики, но тогда доступа из вне к прометеусу не будет)
- Исходящий - разрешаем трафик на все сервисы с которых периодически скрапим метрики, ну и kube-system дефолтно.

Grafana:
- Входящий - также как Prometheus, можете дать доступ обращаться только Prometheus, но тогда доступа, опять же, не будет.
- Исходящий - скрапим Prometheus, и также даем возможность обращаться к kube-system дефолтно за ДНС.
